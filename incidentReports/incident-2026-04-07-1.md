# Incident: 2026-04-07 20:43 UTC

## Summary

Between approximately 20:40 and 20:44 UTC on April 7, 2026, all pizza orders submitted through the JWT Pizza service failed with a 500 status code and the message "Failed to fulfill order at factory." The event was triggered by a chaos injection initiated via the JWT Pizza Factory vendor portal. The factory began returning errors for all order creation requests, causing 100% of pizza orders to fail during the incident window.

The event was detected by manual monitoring of the application. The team resolved the incident by identifying the `reportUrl` in the factory error response and visiting it to disable the chaos. This was a **critical** severity incident affecting all users attempting to place orders.

## Detection

The incident was detected when the on-call engineer noticed order failures during routine monitoring. Grafana alert rules for factory errors and pizza creation failures were configured but the chaos window timing meant the incident required manual observation.

**Improvement:** Alert thresholds and notification configuration were refined during this incident to ensure faster automated detection in the future. Specifically, the Error Log Spike alert was updated to monitor factory-specific errors rather than all application errors.

## Impact

For approximately 4 minutes between 20:40 UTC and 20:44 UTC on 04/07/2026, all users experienced complete inability to place pizza orders. The pizza factory returned errors for every order request during this window.

This incident affected 100% of order-placing users. Menu browsing, authentication, and franchise lookups remained functional throughout.

## Response

The on-call engineer detected the incident through manual monitoring and immediately began diagnosis. A test order was placed via curl, which revealed both the failure mode ("Failed to fulfill order at factory") and the resolution mechanism (`followLinkToEndChaos` URL in the response body). The chaos was resolved within approximately 1 minute of detection.

## Root cause

The JWT Pizza Factory chaos testing system injected a fault that caused the factory to reject all incoming pizza order requests with an error. This was an intentional chaos injection initiated via the vendor portal at pizza-factory.cs329.click.

## Resolution

The service was restored by visiting the `reportUrl` provided in the factory's error response. This URL (`https://pizza-factory.cs329.click/api/support/715529557b314acaa6300c22837ae2a5/report/bii1ga4r`) signaled the factory to stop the chaos injection. Orders immediately resumed processing normally after the URL was visited.

## Prevention

1. Ensure Grafana alert rules are properly configured with correct metric names and thresholds to detect factory failures automatically.
2. Maintain properly configured notification contact points so alerts are delivered promptly.
3. Keep traffic simulation running continuously so that failures generate detectable metrics.

## Action items

1. **Fixed Prometheus metrics pipeline** - Metrics were not being delivered to Grafana Cloud due to an incorrect API key. Updated the access token in GitHub secrets and redeployed. Metrics now flow correctly and enable Prometheus-based alerting.
2. **Updated alert queries** - Corrected metric names in alert rules (added `_milliseconds` suffix for latency metrics, updated `source` label to match deployed config).
3. **Refined Error Log Spike alert** - Changed from monitoring all application errors to specifically monitoring factory errors (`type="factory"`) to reduce false positives.
4. **Set contact points on all alerts** - Ensured all 4 alert rules route notifications to the `JWT Pizza On Call` contact point.
5. **Removed intentional failures from traffic simulation** - Removed the 21-pizza order that deliberately triggered factory failures, which was masking real incident alerts.
